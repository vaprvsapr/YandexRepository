# Event Manager API

RESTful API для управления событиями, написанное на C# с использованием ASP.NET Core (.NET 10) и принципов чистой архитектуры (Dependency Injection, DTO, Repository Pattern).

## Требования

- **.NET 10 SDK** 
- Среда разработки (Visual Studio 2026 / Rider / VS Code)

## Запуск проекта

### Через Visual Studio
1. Откройте решение в Visual Studio.
2. Убедитесь, что `EventManager` установлен как запускаемый проект.
3. Нажмите `F5` или кнопку "Пуск" (IIS Express / http / https).
4. Автоматически откроется браузер со страницей **Swagger UI**, где можно протестировать API.

### Через командную строку (CLI)
1. Откройте терминал в папке проекта `EventManager`.
2. Выполните команду запуска:
   ```bash
   dotnet run
   ```
3. В консоли отобразится URL, на котором запущено приложение (например, `https://localhost:7001`).
4. Перейдите в браузере по адресу `https://localhost:<port>/swagger`, чтобы посмотреть интерактивную документацию Swagger и отправлять запросы.

## Краткая документация API


Базовые пути (Base URL):
- Для событий: `/events`
- Для бронирований: `/bookings` и `/events/{id}/book`

### Эндпоинты

| Метод    | Маршрут                      | Описание                                               | Успешный ответ   | Возможные ошибки                   |
|----------|------------------------------|--------------------------------------------------------|------------------|------------------------------------|
| `GET`    | `/events`                    | Получить коллекцию событий (поддерживает query-параметры) | `200 OK`         | `400 Bad Request`                  |
| `GET`    | `/events/{id}`               | Получить событие по его уникальному `id`               | `200 OK`         | `404 Not Found`                    |
| `POST`   | `/events`                    | Создать новое событие                                  | `201 Created`    | `400 Bad Request`, `409 Conflict`  |
| `PUT`    | `/events/{id}`               | Обновить существующее событие по его `id`              | `200 OK`         | `400 Bad Request`, `404 Not Found` |
| `DELETE` | `/events/{id}`               | Удалить событие по его `id`                            | `204 No Content` | `404 Not Found`                    |
| `POST`   | `/events/{id}/book`          | Создать бронирование для события с указанным `id`      | `202 Accepted`   | `404 Not Found`                    |
| `GET`    | `/bookings/{id}`             | Получить бронирование по его уникальному `id`          | `200 OK`         | `404 Not Found`                    |
| `GET`    | `/bookings/event/{id}`       | Получить список бронирований для события с указанным `id` | `200 OK`         | `404 Not Found`                    |


### Модели данных

#### Event (событие)
```json
{
  "id": "guid",
  "title": "Еженедельное планирование",
  "description": "Обсуждение ключевых задач на следующую неделю",
  "startAt": "2026-05-15T10:00:00Z",
  "endAt": "2026-05-15T11:00:00Z"
}
```
*Дата окончания (`endAt`) должна быть строго больше даты начала (`startAt`).*

#### Booking (бронирование)
```json
{
  "id": "guid",
  "eventId": "guid",
  "status": "Pending | Confirmed | ...",
  "createdAt": "2026-05-15T09:55:00Z",
  "processedAt": "2026-05-15T10:00:00Z"
}
```

### Фильтрация и пагинация

Метод **GET `/events`** поддерживает гибкую систему фильтрации и разбиения на страницы через query-параметры.

#### Параметры фильтрации

| Параметр | Тип | Обязательный | Описание |
|----------|-----|--------------|---------|
| `title` | строка | Нет | Подстроковый поиск по названию события (регистронезависимый). Возвращаются события, название которых содержит эту подстроку. |
| `from` | ISO 8601 datetime | Нет | Начало временного диапазона. Возвращаются события, у которых `startAt >= from` |
| `to` | ISO 8601 datetime | Нет | Конец временного диапазона. Возвращаются события, у которых `endAt <= to` |

#### Параметры пагинации

| Параметр | Тип | Обязательный | Значение по умолчанию | Ограничения | Описание |
|----------|-----|--------------|----------------------|-------------|---------|
| `page` | целое число | Нет | 1 | Минимум: 1 | Номер страницы для отображения. Первая страница — это 1 |
| `pageSize` | целое число | Нет | 10 | От 1 до 100 | Количество элементов на одной странице |

#### Примеры комбинаций параметров

```bash
# Базовый запрос (первая страница, 10 элементов)
GET /events

# Фильтрация по названию
GET /events?title=встреча

# Временной диапазон
GET /events?from=2026-05-01T00:00:00Z&to=2026-05-31T23:59:59Z

# Название + временной диапазон
GET /events?title=встреча&from=2026-05-01T00:00:00Z&to=2026-05-31T23:59:59Z

# Только пагинация
GET /events?page=3&pageSize=20

# Все параметры вместе
GET /events?title=встреча&from=2026-05-01T00:00:00Z&to=2026-05-31T23:59:59Z&page=2&pageSize=5
```

#### Как работает пагинация

Результат пагинирует события в порядке их получения из хранилища:
- Если всего **42 события** и `pageSize=10`:
  - Page 1 вернёт события 1–10
  - Page 2 вернёт события 11–20
  - Page 3 вернёт события 21–30
  - Page 4 вернёт события 31–40
  - Page 5 вернёт события 41–42

Поле `totalCount` в ответе всегда показывает **общее количество событий, удовлетворяющих условиям фильтрации** (независимо от пагинации). Это позволяет клиенту вычислить общее количество страниц: `Math.Ceil(totalCount / pageSize)`.

### Формат ответов и Глобальная обработка исключений

Успешные ответы API возвращают сами данные, например, массив событий или конкретный объект `EventDto`, без дополнительной обертки.

#### Пример ответа 200 OK (GET /events/1)
```json
{
  "id": 1,
  "title": "Еженедельное планирование",
  "description": "Обсуждение ключевых задач на следующую неделю",
  "startAt": "2026-05-15T10:00:00Z",
  "endAt": "2026-05-15T11:00:00Z"
}
```

Все возникающие в API ошибки обрабатываются с помощью **Global Exception Handling Middleware**. Это гарантирует, что клиенты всегда будут получать информацию об ошибках (`400 Bad Request`, `404 Not Found`, `409 Conflict`, `500 Internal Server Error`) в унифицированном и предсказуемом формате по стандарту **RFC 7807 (Problem Details)**.

#### Пример ответа с ошибкой валидации (400 Bad Request)
```json
{
  "type": "https://tools.ietf.org/html/rfc9110#section-15.5.1",
  "title": "One or more validation errors occurred.",
  "status": 400,
  "instance": "/events",
  "errors": {
    "startAt": [
      "Время начала события должно быть меньше времени окончания события."
    ]
  },
  "traceId": "00-cbdfdd3839afff381fc3d46068b1c7ae-f173bc6c6fbb4c73-00"
}
```
Свойство `instance` указывает на маршрут, при обращении к которому возникла ошибка.




## Архитектура и сервисы приложения

### IRepository<T>

IRepository<T> — универсальный обобщённый интерфейс репозитория, используемый для управления коллекциями сущностей любого типа (например, событий или бронирований). Предоставляет стандартные методы для получения объекта по идентификатору, получения всех объектов, добавления, обновления и удаления. Благодаря универсальности интерфейс позволяет реализовать единый подход к работе с разными типами данных, обеспечивая переиспользуемость и простоту тестирования бизнес-логики.

### BookingService

BookingService — сервис, реализующий бизнес-логику для работы с бронированиями событий. Позволяет создавать новые бронирования для событий, получать отдельные бронирования по идентификатору, а также получать список всех бронирований для конкретного события. При создании бронирования сервис проверяет существование события, генерирует уникальный идентификатор брони, устанавливает статус <code>Pending</code> и сохраняет дату создания и обработки. Все методы выбрасывают исключения при ошибках (например, если событие или бронирование не найдено).

### EventService

EventService — сервис, реализующий бизнес-логику для управления событиями. Поддерживает создание, удаление, обновление, получение одного события и получение списка событий с фильтрацией и пагинацией. При создании события проверяет уникальность идентификатора, при обновлении и удалении — существование события. Для получения списка событий поддерживает фильтрацию по названию, диапазону дат, а также постраничный вывод (пагинацию).

### BookingProcessingService

BookingProcessingService — это фоновый сервис, который автоматически подтверждает новые бронирования.

- Периодически (раз в 5 секунд) ищет бронирования со статусом <code>Pending</code>.
- Подтверждает их (устанавливает статус <code>Confirmed</code> и обновляет <code>processedAt</code>).
- Ведёт логирование этапов обработки через встроенный ILogger.
- Работает до остановки приложения.

Это позволяет реализовать асинхронную обработку бронирований и отделить момент создания брони от её подтверждения. Сервис не требует ручного запуска и стартует вместе с приложением.

## Детальное описание методов

### GET `/events`
Получение списка всех событий с поддержкой фильтрации и пагинации. Параметры передаются как query-параметры в URL.

**Поддерживаемые параметры фильтрации:**
- `title` (строка, опционально) — фильтр по названию события (подстроковый поиск, регистронезависимый)
- `from` (ISO 8601 datetime, опционально) — фильтр по начальной дате. Возвращает события, начинающиеся в этот момент или позже
- `to` (ISO 8601 datetime, опционально) — фильтр по конечной дате. Возвращает события, заканчивающиеся в этот момент или раньше
- `page` (целое число, опционально, по умолчанию 1) — номер страницы для пагинации. Минимальное значение: 1
- `pageSize` (целое число, опционально, по умолчанию 10) — количество элементов на одной странице. Диапазон значений: 1–100

**Статус-коды:**
- **200 OK**: Успешное выполнение. Возвращает объект `PaginatedResultDto` со списком событий, общим количеством и информацией о пагинации
- **400 Bad Request**: Некорректные параметры запроса (например, `page <= 0`, `pageSize > 100`, `from > to`)

**Примеры запросов:**

1. Получить все события (первая страница по умолчанию):
```bash
curl -X GET "https://localhost:7096/events" \
  -H "accept: application/json"
```

2. Получить события, отфильтрованные по названию:
```bash
curl -X GET "https://localhost:7096/events?title=планирование" \
  -H "accept: application/json"
```

3. Получить события в определённом диапазоне времени:
```bash
curl -X GET "https://localhost:7096/events?from=2026-05-15T00:00:00Z&to=2026-05-20T23:59:59Z" \
  -H "accept: application/json"
```

4. Комбинированная фильтрация: название + временной диапазон + пагинация:
```bash
curl -X GET "https://localhost:7096/events?title=встреча&from=2026-05-01T00:00:00Z&to=2026-05-31T23:59:59Z&page=2&pageSize=5" \
  -H "accept: application/json"
```

5. Получить вторую страницу с 20 элементами на странице:
```bash
curl -X GET "https://localhost:7096/events?page=2&pageSize=20" \
  -H "accept: application/json"
```

**Пример ответа 200 OK:**
```json
{
  "events": [
    {
      "id": 1,
      "title": "Еженедельное планирование",
      "description": "Обсуждение ключевых задач на следующую неделю",
      "startAt": "2026-05-15T10:00:00Z",
      "endAt": "2026-05-15T11:00:00Z"
    },
    {
      "id": 2,
      "title": "Встреча с командой",
      "description": "Синхронизация по текущим задачам",
      "startAt": "2026-05-15T14:00:00Z",
      "endAt": "2026-05-15T15:00:00Z"
    }
  ],
  "totalCount": 42,
  "pageSize": 10,
  "page": 1
}
```

**Пример ответа 400 Bad Request (неверные параметры пагинации):**
```json
{
  "type": "https://tools.ietf.org/html/rfc9110#section-15.5.1",
  "title": "One or more validation errors occurred.",
  "status": 400,
  "instance": "/events",
  "errors": {
    "pageSize": [
      "Количество элементов на странице должно быть между 1 и 100."
    ]
  },
  "traceId": "00-cbdfdd3839afff381fc3d46068b1c7ae-f173bc6c6fbb4c73-00"
}
```

### POST `/events`
Создание нового события. В теле запроса передаётся JSON-объект с данными события.

**Параметры (в теле запроса):**
- `id` (целое число, обязательно) — уникальный идентификатор события
- `title` (строка, обязательно) — название события, не должно быть пустой строкой
- `description` (строка, опционально) — описание события, может быть пустой строкой
- `startAt` (ISO 8601 datetime, обязательно) — дата и время начала события
- `endAt` (ISO 8601 datetime, обязательно) — дата и время окончания события. **ВАЖНО:** должна быть строго больше `startAt`

**Статус-коды:**
- **201 Created**: Успешное создание события. Возвращает созданный объект `EventDto` и заголовок `Location` со ссылкой на маршрут получения события
- **400 Bad Request**: Ошибка валидации:
  - Пропущено обязательное поле (`id`, `title`, `startAt`, `endAt`)
  - `endAt` меньше или равен `startAt`
  - `title` — пустая строка
- **409 Conflict**: Событие с таким ID уже существует в системе

**Примеры запросов:**

1. Создать простое событие:
```bash
curl -X POST "https://localhost:7096/events" \
  -H "Content-Type: application/json" \
  -H "accept: application/json" \
  -d '{
    "id": 1,
    "title": "Еженедельное планирование",
    "description": "Обсуждение ключевых задач на следующую неделю",
    "startAt": "2026-05-15T10:00:00Z",
    "endAt": "2026-05-15T11:00:00Z"
  }'
```

2. Создать событие без описания:
```bash
curl -X POST "https://localhost:7096/events" \
  -H "Content-Type: application/json" \
  -H "accept: application/json" \
  -d '{
    "id": 2,
    "title": "Встреча с клиентом",
    "description": "",
    "startAt": "2026-05-16T09:00:00Z",
    "endAt": "2026-05-16T10:00:00Z"
  }'
```

**Пример ответа 201 Created:**
```json
{
  "id": 1,
  "title": "Еженедельное планирование",
  "description": "Обсуждение ключевых задач на следующую неделю",
  "startAt": "2026-05-15T10:00:00Z",
  "endAt": "2026-05-15T11:00:00Z"
}
```

**Пример ответа 400 Bad Request (endAt раньше startAt):**
```json
{
  "type": "https://tools.ietf.org/html/rfc9110#section-15.5.1",
  "title": "One or more validation errors occurred.",
  "status": 400,
  "instance": "/events",
  "errors": {
    "endAt": [
      "Время окончания события должно быть больше времени начала события."
    ]
  },
  "traceId": "00-cbdfdd3839afff381fc3d46068b1c7ae-f173bc6c6fbb4c73-00"
}
```

**Пример ответа 409 Conflict (событие с таким ID существует):**
```json
{
  "type": "InvalidOperationException",
  "title": "Событие с id 1 уже существует.",
  "status": 409,
  "detail": "Class: EventManager.Services, Method: CreateEvent",
  "traceId": "00-cbdfdd3839afff381fc3d46068b1c7ae-f173bc6c6fbb4c73-00"
}
```

### PUT `/events/{id}`
Обновление данных существующего события. Идентификатор передаётся в URL, изменяемые поля — в теле запроса (JSON).

**Параметры:**
- `id` (целое число, обязательно, в URL) — идентификатор события для обновления

**Параметры в теле запроса (все опциональны):**
- `title` (строка, опционально) — новое название события
- `description` (строка, опционально) — новое описание события
- `startAt` (ISO 8601 datetime, опционально) — новая дата и время начала события
- `endAt` (ISO 8601 datetime, опционально) — новая дата и время окончания события

**Правила валидации:**
- Если `startAt` и `endAt` переданы вместе, то `endAt` должна быть больше `startAt`
- Если поле не передано, оно сохранит старое значение
- Все поля, которые передаются, должны пройти валидацию

**Статус-коды:**
- **200 OK**: Обновление прошло успешно
- **400 Bad Request**: Ошибка валидации (например, `endAt < startAt`)
- **404 Not Found**: Событие с указанным `id` не найдено

**Примеры запросов:**

1. Обновить только название события:
```bash
curl -X PUT "https://localhost:7096/events/1" \
  -H "Content-Type: application/json" \
  -H "accept: application/json" \
  -d '{
```

### DELETE `/events/{id}`
Удаление события из системы по его `id`.

**Параметры:**
- `id` (целое число, обязательно) — уникальный идентификатор события для удаления

**Статус-коды:**
- **204 No Content**: Удаление прошло успешно (ответ не содержит тела)
- **404 Not Found**: Попытка удалить несуществующее событие

**Примеры запросов:**

```bash
curl -X DELETE "https://localhost:7096/events/1" \
```

```bash
curl -X DELETE "https://localhost:7096/events/42" \
```

### POST `/events/{id}/book`
Создаёт новое бронирование для события с указанным идентификатором.

**Параметры:**
- `id` (GUID, в URL) — идентификатор события

**Статус-коды:**
- **202 Accepted**: Бронирование принято к обработке. В теле ответа — созданный объект BookingDto, в заголовке Location — ссылка на бронирование.

```bash
curl -X POST "https://localhost:7096/events/1/book" -H "accept: application/json"
```

**Пример ответа 202 Accepted:**
```json
{
  "id": "guid",
  "eventId": "guid",
  "status": "Pending"
}
```
