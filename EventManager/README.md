# Event Manager API

RESTful API для управления событиями, написанное на C# с использованием ASP.NET Core (.NET 10) и принципов чистой архитектуры (Dependency Injection, DTO, Repository Pattern).

## Требования

- **.NET 10 SDK** 
- Среда разработки (Visual Studio 2026 / Rider / VS Code)

## Запуск проекта

### Через Visual Studio
1. Откройте решение в Visual Studio.
2. Убедитесь, что `EventManager` установлен как запускаемый проект.
3. Нажмите `F5` или кнопку "Пуск" (IIS Express / http / https).
4. Автоматически откроется браузер со страницей **Swagger UI**, где можно протестировать API.

### Через командную строку (CLI)
1. Откройте терминал в папке проекта `EventManager`.
2. Выполните команду запуска:
   ```bash
   dotnet run
   ```
3. В консоли отобразится URL, на котором запущено приложение (например, `https://localhost:7001`).
4. Перейдите в браузере по адресу `https://localhost:<port>/swagger`, чтобы посмотреть интерактивную документацию Swagger и отправлять запросы.

## Краткая документация API

Базовый путь (Base URL) для всех запросов: `/events`

### Эндпоинты

| Метод    | Маршрут            | Описание                                  | Успешный ответ   | Возможные ошибки                   |
|----------|--------------------|-------------------------------------------|------------------|------------------------------------|
| `GET`    | `/events`          | Получить коллекцию событий (поддерживает query-параметры) | `200 OK`         | `400 Bad Request`                  |
| `GET`    | `/events/{id}`     | Получить событие по его уникальному `id`  | `200 OK`         | `404 Not Found`                    |
| `POST`   | `/events`          | Создать новое событие                     | `201 Created`    | `400 Bad Request`, `409 Conflict`  |
| `PUT`    | `/events/{id}`     | Обновить существующее событие по его `id` | `200 OK`         | `400 Bad Request`, `404 Not Found` |
| `DELETE` | `/events/{id}`     | Удалить событие по его `id`               | `204 No Content` | `404 Not Found`                    |

### Детальное описание методов

#### GET `/events`
Получение списка всех событий. Обрабатывает входящие `query`-параметры (например, для фильтрации или разбиения на страницы).
- **200 OK**: Успешное выполнение. Возвращает массив объектов событий в формате JSON.
- **400 Bad Request**: Некорректные параметры запроса.

#### GET `/events/{id}`
Получение конкретного события по его уникальному целочисленному идентификатору `id`.
- **200 OK**: Событие найдено. Возвращает объект `EventDto`.
- **404 Not Found**: Событие с таким идентификатором отсутствует в системе.

#### POST `/events`
Создание нового события. В теле запроса ожидается JSON-объект с данными.
- **201 Created**: Успешное создание. Возвращает созданный объект `EventDto` и содержит заголовок `Location` со ссылкой на маршрут для получения события.
- **400 Bad Request**: Ошибка валидации (например, пропущено обязательное поле, либо дата окончания установлена раньше даты начала).
- **409 Conflict**: Конфликт на уровне бизнес-логики при попытке создать событие.

#### PUT `/events/{id}`
Обновление данных существующего события. Идентификатор передаётся в URL, а изменения — в теле запроса (JSON).
- **200 OK**: Обновление прошло успешно. Приложение возвращает пустой ответ или обновлённые данные (зависит от реализации).
- **400 Bad Request**: Ошибка валидации переданных данных на обновление.
- **404 Not Found**: Событие с указанным `id` не найдено.

#### DELETE `/events/{id}`
Удаление события из системы по его `id`.
- **204 No Content**: Удаление прошло успешно (ответ не содержит тела).
- **404 Not Found**: Попытка удалить несуществующее событие.

### Модель данных для создания и обновления (JSON)

Обратите внимание: дата окончания (`endAt`) должна быть строго больше даты начала (`startAt`).

```json
{
  "id": 1,
  "title": "Еженедельное планирование",
  "description": "Обсуждение ключевых задач на следующую неделю",
  "startAt": "2026-05-15T10:00:00Z",
  "endAt": "2026-05-15T11:00:00Z"
}
```

### Формат ответов и Глобальная обработка исключений

Успешные ответы API возвращают сами данные, например, массив событий или конкретный объект `EventDto`, без дополнительной обертки.

#### Пример ответа 200 OK (GET /events/1)
```json
{
  "id": 1,
  "title": "Еженедельное планирование",
  "description": "Обсуждение ключевых задач на следующую неделю",
  "startAt": "2026-05-15T10:00:00Z",
  "endAt": "2026-05-15T11:00:00Z"
}
```

Все возникающие в API ошибки обрабатываются с помощью **Global Exception Handling Middleware**. Это гарантирует, что клиенты всегда будут получать информацию об ошибках (`400 Bad Request`, `404 Not Found`, `409 Conflict`, `500 Internal Server Error`) в унифицированном и предсказуемом формате по стандарту **RFC 7807 (Problem Details)**.

#### Пример ответа с ошибкой валидации (400 Bad Request)
```json
{
  "type": "https://tools.ietf.org/html/rfc9110#section-15.5.1",
  "title": "One or more validation errors occurred.",
  "status": 400,
  "instance": "/events",
  "errors": {
    "startAt": [
      "Время начала события должно быть меньше времени окончания события."
    ]
  },
  "traceId": "00-cbdfdd3839afff381fc3d46068b1c7ae-f173bc6c6fbb4c73-00"
}
```
Свойство `instance` указывает на маршрут, при обращении к которому возникла ошибка.

## Тестирование

В решении присутствует проект с модульными тестами `EventManager.Tests`, написанный для проверки бизнес-логики и работы API. Проект использует современные инструменты .NET 10:
- **xUnit** — фреймворк для запуска тестов.
- **Moq** — библиотека для создания заглушек (mock-объектов) для изоляции тестируемых компонентов.
- **coverlet.collector** — инструмент для сбора метрик покрытия кода (Code Coverage).

### Запуск тестов

Вы можете запустить тесты через Test Explorer в Visual Studio или через интерфейс командной строки (CLI).
Для запуска тестов через CLI перейдите в корневую папку решения или проекта `EventManager.Tests` и выполните команду:
```bash
dotnet test
```
